# ADR-003: API Entity Adapter Pattern

## Status
Accepted

## Context
In a client-server architecture, database changes and API endpoint adjustments are common. Leakage of backend schemas (Data Transfer Objects or DTOs) directly into UI components creates a tight coupling. If a field name is changed (e.g., `smartAccount` to `accountAddress`), developers must update code across multiple components, hooks, and tests, increasing the risk of regression.

## Decision
We implemented the **Adapter Pattern** for all server communications. 

### Enforcements
* **Isolation Layer**: Every entity slice must implement an adapter file inside `src/entities/*/model/adapter.ts` (e.g., `toPortfolio()`, `toTransaction()`, `toSessionKey()`).
* **Conversion Boundary**: The API client fetches raw DTOs, but the React Query hook maps the response through the corresponding entity adapter *before* returning the data to the UI.
* **Casting & Formatting**: Adapters must normalize casing (e.g., snake_case to camelCase), parse raw date strings into JS `Date` objects, and cast blockchain-centric formats (such as converting stringified values into `BigInt` or formatted decimals).

```text
Backend DTO -> Entity Adapter -> Type-safe Frontend Model -> React Component
```

## Alternatives Considered
* **Shared TypeScript Typings**: Generating shared typings from the backend controllers. However, this only solves type safety, not runtime conversions or logical normalization of models.
* **Direct DTO Consumption**: Consuming backend schemas as-is. Rejected because it exposes the UI layer to backend schema volatility.

## Consequences
* **Positives**:
  * Decouples visual elements from server-side database schema changes.
  * Cleans up UI code by shifting date parsing and math conversion logic out of render cycles.
  * Simplifies unit testing of components since frontend models are clean, logical representations.
* **Negatives**:
  * Introduces an extra mapping layer with additional files and types.
  * Developers must write matching frontend type definitions parallel to backend schemas.

## Future Considerations
Develop automated contract alignment testing in the CI to confirm that adapter inputs are compatible with the latest swagger/OpenAPI schemas generated by the backend.
